Capítulo 09

Integrações: Gmail, Google Drive e Calendário

Seu agente só é verdadeiramente útil quando conectado às ferramentas que você já usa. Neste capítulo, vamos conectar o agente ao Gmail, Google Drive e Google Calendar.

Por Que Google Workspace?

A maioria das pessoas já usa Gmail e Google Drive. Conectar seu agente a essas ferramentas significa que ele pode:

  • 📧 Ler, resumir e responder e-mails automaticamente
  • 📁 Salvar documentos e relatórios no Drive
  • 📅 Verificar sua agenda e criar eventos
  • 📊 Ler e escrever em planilhas do Sheets
  • 📝 Criar e editar documentos no Docs

Usando o GOG CLI (Google Workspace CLI)

💡
O que é GOG? GOG (Google Operations Gateway) é uma ferramenta de linha de comando que conecta o OpenClaw ao Google Workspace. É como uma "ponte" entre seu agente e todos os serviços do Google.
# Instalar o GOG CLI npm install -g gog-cli # Verificar instalação gog --version

Configurando a Service Account do Google

ℹ️
O que é Service Account? É uma "conta de serviço" criada no Google Cloud que permite que programas (como seu agente) acessem os serviços do Google sem precisar digitar sua senha toda vez. É como uma chave mestra para o seu Google Workspace.
1

Acessar o Google Cloud Console

Vá em console.cloud.google.com, faça login com sua conta Google e crie um novo projeto (ex: "Meu Agente IA").

2

Ativar as APIs necessárias

No menu, vá em APIs & Services → Enable APIs. Ative: Gmail API, Google Drive API, Google Calendar API, Google Sheets API.

3

Criar Service Account

Em IAM & Admin → Service Accounts → Create Service Account. Dê um nome e clique em "Create and Continue".

4

Baixar a chave JSON

Clique na service account criada → Keys → Add Key → Create New Key → JSON. Vai baixar um arquivo .json. Faça upload na VPS.

# Copiar o arquivo de chave para a VPS (do seu computador) # Execute esse comando no seu computador local, não na VPS scp ~/Downloads/meu-projeto-xxxxx.json root@SEU_IP_VPS:~/.openclaw/workspace/google-sa.json # Na VPS: testar a autenticação GOG_ACCOUNT=seu@gmail.com gog gmail labels list

Exemplos de Comandos com Gmail

# Ler e-mails não lidos dos últimos 7 dias gog gmail search 'is:unread newer_than:7d' # Enviar e-mail gog gmail send \ --to "cliente@email.com" \ --subject "Retorno ao seu contato" \ --body "Olá! Vi sua mensagem e..." # Listar eventos do calendário de hoje gog calendar events list --today # Criar evento no calendário gog calendar events create \ --summary "Reunião de Vendas" \ --start "2026-04-01T15:00:00-03:00" \ --end "2026-04-01T16:00:00-03:00" # Fazer upload de arquivo no Drive gog drive upload ~/.openclaw/workspace/relatorio.pdf
⚠️
Delegação de Domínio: Para o agente acessar Gmail e Calendar em seu nome, você precisa ativar a "Delegação de Domínio" na Service Account. Isso só funciona com Google Workspace (conta paga). Para Gmail pessoal (@gmail.com), use OAuth2 normal.

🗄️ Supabase — A Integração Principal para Persistência

💡
O que é o Supabase? Supabase é um banco de dados PostgreSQL na nuvem com API automática, autenticação e Realtime integrados. Para seus agentes, é o lugar onde toda atividade fica registrada — tarefas, conhecimento, status, métricas — tudo acessível em tempo real no dashboard.

Enquanto o Google Workspace serve para comunicação (e-mail, calendário, documentos), o Supabase é a espinha dorsal de dados do seu time de agentes: onde tudo é registrado, consultado e sincronizado em tempo real.

Tabelas Principais do seu projeto

Tabela O Que Armazena Quem Usa
tasks Código, título, status, agente responsável, timestamps Todos os agentes — toda tarefa registrada aqui
knowledge Base de conhecimento indexável (código + conteúdo) Busca semântica, consultas frequentes
agent_status Estado de cada agente: working/standby/idle/blocked Monitor de agentes (cron 15 min)
daily_metrics Custos de LLM, tokens usados, tarefas concluídas por dia Dashboard de custos

Configuração Básica

# Registrar tarefa no início (status: in_progress) SVC="SUA_SERVICE_ROLE_KEY" NOW=$(date -u +"%Y-%m-%dT%H:%M:%SZ") curl -s -X POST "https://SEU_PROJECT.supabase.co/rest/v1/tasks" \ -H "Authorization: Bearer $SVC" \ -H "apikey: $SVC" \ -H "Content-Type: application/json" \ -H "Prefer: return=representation" \ -d "{ \"title\": \"Minha tarefa\", \"description\": \"O que vou fazer\", \"agent_id\": \"hefesto\", \"status\": \"in_progress\", \"vertical\": \"tecnologia\" }" # Marcar como concluída curl -s -X PATCH "https://SEU_PROJECT.supabase.co/rest/v1/tasks?code=eq.TSK-001" \ -H "Authorization: Bearer $SVC" \ -H "apikey: $SVC" \ -H "Content-Type: application/json" \ -d "{\"status\":\"done\",\"completed_at\":\"$NOW\"}"
🚨
Regra de segurança RLS: Use sempre a service_role key para escrita (INSERT/UPDATE) nos agentes. A anon key é só para leitura. Nunca exponha a service_role key no frontend ou em repositórios públicos.
ℹ️
Supabase Realtime: Quando um agente faz PATCH no banco (ex: status: done), o dashboard atualiza automaticamente sem reload — graças ao Supabase Realtime via WebSocket. É o que faz o dashboard parecer vivo.
